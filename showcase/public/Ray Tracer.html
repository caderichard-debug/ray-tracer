Abstract

Abstract.

Interactive CPU ray tracing is a rare case where the memory hierarchy, the vector units, and the threading model are all visible in the hot loop. Writing one from scratch in C++17 makes you pay for every decision in wall-clock, which is the entire point.

This project puts every optimization behind a compile-time gate — ENABLE_AVX, ENABLE_OPENMP, ENABLE_MORTON, ENABLE_LOOP_UNROLL, and a dozen others — so the benchmark harness can cleanly attribute each ms. The scene is small, the numbers are honest, and the places where a textbook optimization fails are published alongside the places where it wins.

Under the hood: a BVH with surface-area heuristic construction; handwritten AVX2 ray-packet traversal over 8 rays at a time; a wavefront tile scheduler; Z-order (Morton) pixel scheduling for cache locality; stratified, progressive, and variance-driven adaptive sampling; a per-thread PCG random number generator; dielectric, metal, lambertian and emissive materials; thin-lens depth-of-field; and a deterministic regression harness that SHA-compares output PPMs to golden images.

What you won't find: full unbiased global illumination, multiple-importance sampling, caustics. This is a direct-lighting-plus-recursion tracer — on purpose, so the scope stayed finite.

What's in the engine

Twelve switches.

Some don't show meaningful performance improvements on a the small 25-primitive test scene.

01 / acceleration

SAH BVH

Broad-phase for spheres & triangles, also used for any-hit shadow queries. Built with a surface-area-heuristic split cost.

src/acceleration/bvh.cpp
02 / SIMD

AVX2 ray packets

8 rays (4×2 block) traced in lockstep over SoA sphere data. Handwritten AVX2 helpers for Color/Vec3 in shading and framebuffer finalize.

sphere_simd.h · renderer.cpp
+2.5× vs scalar
03 / scheduling

Wavefront tiles

Rays are batched into waves of tiles, keeping shading data hot in L1/L2 as the tile migrates across cores.

renderer.cpp · enable_wavefront
+3.1× vs scalar
04 / locality

Morton (Z-order)

Pixel traversal follows a space-filling curve instead of scanlines, so neighbouring pixels land near each other in the accumulation buffer.

Morton::generate_morton_pixels
+2.0× vs scalar
05 / threading

OpenMP

Pragma-based parallel-for on the pixel loop; ~1.95× going 1 → 8 threads on a 1280×720 scalar sweep. Determinism preserved by turning OMP off for the regression harness.

Makefile · ENABLE_OPENMP
+1.95× at 8 threads
06 / sampling

Stratified AA

Jittered grid over the pixel footprint, so AA samples stop clumping. Roughly 19% faster convergence at the same spp vs. white-noise.

main.cpp · ENABLE_STRATIFIED
−19% time @ same spp
07 / sampling

Adaptive (variance-driven)

Spends samples where variance demands them; finishes flat regions early. ~72% less time at half the primary samples — caveat: that's fewer rays, not a pure speedup.

renderer.cpp · enable_adaptive
−72% time (½ primaries)
08 / sampling

Progressive

Multi-pass accumulation with a flat row-major buffer (cache-friendlier than a nested vector<vector>). First pixels land in ~30 ms.

renderer.cpp · enable_progressive
09 / rng

PCG, per-thread

A fast, well-distributed PCG generator with no locks; the sampling hot path calls it directly. ENABLE_FAST_RNG swaps it for rand() as a baseline.

src/math/pcg_random.h
10 / acceleration

BVH overhead

At 25 primitives, BVH and frustum culling come in slower than brute-force scalar — the traversal stack costs more than it saves. Wins appear on the bigger scenes in tests/.

benchmark_results/ · full log
11 / materials

Materials & light

Lambertian (with textured albedo), metal with fuzz, dielectric with Fresnel, and emissive. Phong shading per light; shadow rays reuse the BVH; N·L ≤ 0 early-out.

material.h · renderer.cpp
12 / correctness

Deterministic regression

The harness renders a fixed Cornell frame with OMP off, hashes the PPM, and fails CI if the SHA drifts. Performance is nothing without a stable bit-exact output to compare against.

tests/regression/

What the slow bars are saying.

BVH and view-frustum culling are slower than brute-force scalar on this scene. That's not a bug — it's scale. Building and descending the tree costs more than it saves when you only have 25 primitives. On a larger scene the crossover reverses; the micro-bench just makes the overhead visible.

The honest summary: SIMD packets, wavefront scheduling, and Morton traversal pay off everywhere. BVH pays off only when the scene is big enough to hide the traversal stack.

Build & run

Build & run.

Everything is gated by Makefile variables. Pick a row, rebuild, rerun the harness — the delta lands in benchmark_results/runs/.

zsh ~/src/ray-tracerfastest build
# full pipeline, 8 threads
$ git clone github.com/caderichard-debug/ray-tracer
$ cd ray-tracer
$ make ENABLE_AVX=1 ENABLE_OPENMP=1 ENABLE_MORTON=1 ENABLE_WAVEFRONT=1 -j
  built ./raytracer (AVX2 · OMP · wavefront)

$ ./raytracer --scene cornell -w 1920 -h 1080 -s 64 -o out.ppm
  rendered 1920×1080 @ 64spp in 221 ms

# reproducible benchmark sweep
$ make bench-feature-deltas
  → benchmark_results/runs/feature_deltas_20260417_175153.md

AVX2 + FMA — 8-ray packet tracing, SIMD color ops.

ENABLE_OPENMP
Parallel-for on the pixel loop. Turn off for deterministic runs.
default ON
ENABLE_PTHREADS
Alternate threading path, mutually exclusive with OMP.
off

Z-order pixel scheduling (batch path).
default ON

Tiled wave scheduler. Fastest path on this machine.
default ON

Jittered grid over pixel footprint for AA.
off

Manual 4× unroll of the inner sample accumulation loop.
off
